专业 / 年级：智能感知工程 / 2024级

这次作业我最终完成的是一个带有场景切换和多重交互的个人主页，而不是传统的“长页面向下滚动式”简历网站。我给它起的整体概念是“个人咖啡馆”：首页不是直接把所有信息铺开，而是先让访问者站在咖啡馆门外，看见 Helena Wang 的名字、专业和一小段关于咖啡的自我介绍，再通过点击门的方式进入室内。这样做的核心原因，是我不希望这个作品看起来像科研项目展示页、课程大作业模板页，或者普通信息卡片堆叠页。我更希望它像一个可以“走进去”的空间，一个带有情绪、带有叙事、也带有个人痕迹的小型数字场景。

在视觉方向上，我没有选择常见的极简白底个人主页，也没有走科技炫光风，而是把整体风格定在“复古咖啡馆 + 个人数据展柜”之间：木质、暖色、窗框、招牌、地图、留声机、咖啡杯、便签、吧台、扶手椅，这些元素共同构成了一个较完整的空间感。它既不是单纯的写实咖啡馆，也不是完全卡通，而是一种介于几何化场景与插画化道具之间的网页表达。这样既能保留前端实现的可控性，又能让页面看上去更有作品感。

## 二、设计说明：我想呈现什么样的自己

这次作品想呈现的，不是一个“把履历罗列出来的人”，而是一个可以通过兴趣、旅行、音乐和情绪空间去被认识的人。我希望观众进入页面时，首先感受到的是氛围，然后才是信息。也就是说，我想让“我是怎样的人”先通过场景被传递，再通过数据和交互被细化。

首页的门面相当于一个自我介绍的前厅。左侧的个人信息卡直接给出名字、专业和一句带情绪温度的咖啡表达：“总有那么一刻，当我捧着咖啡，心怀感恩，正是在这一刻让人觉得神圣。”这句话实际上帮助整个网页确定了气质：它不是嘈杂的，不是强展示欲的，而是有一点安静、有一点感受力、有一点私人收藏馆意味的。

室内部分则承担“数据可视化区”和“交互彩蛋区”的任务。地图对应旅行足迹，是我选择的第一份个人数据。原因在于，相比单纯的作息表、课表打分或者步数曲线，旅行城市更容易让人理解“我眼中的世界是什么样子”。我没有只停留在“去过哪些地方”这种列表层面，而是为每个城市配了一句主观印象。例如东京的雪与温泉、纽约的脚手架与高楼、巴黎的油画和香水、罗马里融化得太快的冰淇淋，这些句子会让数据从“信息”变成“经验”，也更符合老师在作业里强调的“整体能让人记住你”。

留声机对应我的歌单数据。这部分选择了 Top 20 歌单，是因为音乐比一般的客观数据更能体现个体气质。歌单里既有古典电影配乐，也有摇滚、民谣、流行和华语歌曲，这种混搭恰好体现了我审美上并不单线。我没有把这部分做成冷冰冰的表格，而是放在留声机上，让它成为一个“点开后响起记忆”的对象。每首歌后面加了可跳转的播放按钮，也让数据从展示变成可继续探索的入口。

咖啡幸运便签则承担了交互彩蛋的角色。最开始我想做的是“小票机”或“咖啡参数控制台”，但后来觉得这些虽然有趣，却不够自然地贴合当前场景。最终改成“幸运便签”之后，网页的情绪就更统一了：用户点击吧台附近的咖啡区域，会随机抽到一款咖啡，看到它的配比和寄语。这相当于把“数据”和“情绪”合成在一起，既符合咖啡馆主题，也有轻微的仪式感。

## 三、页面结构与功能实现

整个项目由三部分核心体验组成。

第一部分是首页外观页。  
用户初始看到的是 Helena Wang 的咖啡馆门面。左侧是个人信息卡，右侧是咖啡馆建筑，包含门、窗、招牌与装饰。点击门后进入室内。这一层的重点不是信息密度，而是“进入感”。从网页交互上看，这一步其实很简单，但它给整个作品建立了一种过渡：不是浏览页面，而是走进空间。

第二部分是室内数据页。  
地图挂在左上区域，点击后弹出放大视图，同时右侧出现旅行足迹清单；留声机放在右侧，点击后弹出歌单抽屉，显示我的 Top 20 听歌排行，并支持跳转到网易云对应播放界面；下半部分是咖啡吧台和场景道具，点击吧台区域会弹出幸运便签。地图、留声机、咖啡吧台各自承担不同的数据或彩蛋任务，避免所有内容挤在一个单一容器里。

第三部分是多弹窗交互。  
地图弹窗是宽版布局，左边放大图片，右边放城市印象列表；歌单弹窗保留为侧边抽屉样式，突出“榜单感”；咖啡便签则是纸质手写便签风格，强调“抽签”的轻巧感。三个弹窗风格不同，但都在同一个咖啡馆世界观里运作。这一点其实很重要，因为它能避免页面出现“功能有了，但像临时拼上去”的割裂感。

## 四、关键 Prompt 记录

这次作业里，我不是一次把全部需求丢给 AI，而是不断通过“提出方向—看结果—纠偏—再细化”的方式推进。下面摘录几条对项目走向影响最大的关键 Prompt，并附上我当时的修改思路。

### Prompt 1：确定整体概念

直接摘录：

> 我不想做一整个页面一直往下拉的这种，太像科研项目展示了。我希望弄成一个“个人咖啡馆”，第一个页面就显示基础个人信息和简介，然后放一个咖啡馆外观和门，然后点击门可以进入下一个页面，下一个页面是咖啡馆内部，可以看到一张世界地图（点击地图交互之后会放我去过的地方那部分数据），然后旁边放一个留声机（交互之后是歌单数据界面），那个小票彩蛋我再思考一下。先做成这样

AI 输出效果：
从普通长页面改为双场景结构：门外页 + 室内页。

我的修改思路：
这是整个项目最关键的一次方向修正。因为在这之前，网页更像一个高完成度静态展示页，但没有“进入感”。这条 Prompt 相当于把项目从“个人主页”推进成“可探索的个人空间”，也让后面所有素材和交互都有了统一主题。

### Prompt 2：要求真实物件替换抽象 CSS 形状

直接摘录：

> 留声机和地图什么的都变形了，不像留声机。你重新生成图像。

AI 输出效果：
开始使用生成图片资产，而不是仅靠 CSS 画图拼形状。

我的修改思路：
这一轮让我明确意识到：如果一个视觉对象本身承载交互，那么它就不能只“差不多像”。尤其留声机这种辨识度很高的物件，如果做得抽象，整个页面的可信度会被拖低。所以我主动让 AI 进入“生成资产—嵌入页面”的路线，而不是继续在 CSS 形状上修修补补。

### Prompt 3：把歌单数据变成真实内容

直接摘录：

> 下面是歌单数据和对应的网易云音乐链接，你可以在表格/榜单后面放一个小的播放按钮，做网页外链

AI 输出效果：
歌单区域从占位内容变成真实 Top 20 数据，并且附上播放按钮外链。

我的修改思路：
这一步让作业从“概念完整”走向“内容真实”。老师在作业里特别强调数据必须真实属于自己，所以我没有停留在默认歌单，而是直接给出真实歌曲和链接。这样不只是为了完成作业要求，也是为了让作品更像“我的网站”。

### Prompt 4：明确旅行数据的表达方式

直接摘录：

> 首先把地图的贴图从world map换成my_world_map，然后做一个点击可放大显示图片的功能。显示图片的同时右边显示去过的地方列表，包括东京，纽约，洛杉矶，巴黎，柏林，奥克兰，伦敦，罗马。每个地方附上一句评价……

AI 输出效果：
地图部分改成“放大图片 + 右侧印象清单”的结构。

我的修改思路：
这一条 Prompt 解决的是“有数据，但没有我”的问题。单独展示城市名太平，只有配上我对城市的主观印象，这份数据才会真正呈现个人风格。也正是这一步，使地图部分成为整个网页最有人味的一块内容。

### Prompt 5：把幸运抽签和主题真正绑定

直接摘录：

> 幸运抽签部分采用如下素材随机数种子抽取。可以做成一个小便签或者清单，上面可以采用手写体。

AI 输出效果：
咖啡彩蛋从普通弹窗变成带手写感的幸运便签。

我的修改思路：
我很在意“交互彩蛋不能只是多一个功能”。如果它和主题不一致，就会显得是硬塞进去的作业项。将其改成便签之后，它终于像咖啡馆里会出现的东西了，也和“咖啡寄语”形成了自然联系。

## 五、踩坑复盘

这次项目最典型的坑，不在代码报错，而在“效果不对劲”。也就是说，AI 并不是完全做不出来，而是常常能做出一个能运行、却不是我想要的东西。如果我只看“有没有代码”，这个项目早就可以收工；但如果按照作品质量要求来判断，它其实经历了很多轮推翻和重做。

第一个坑是页面结构。  
最开始的方案是单页往下滚动，这其实是 AI 很自然会给出的解法，因为它最稳、最标准、最容易完成。但这个方向不符合我对成品气质的判断。我想要的不是“内容容器”，而是“空间叙事”。这里我学到的是：当我觉得结果“太像模板”时，不能只让 AI 再美化一下，而是应该直接换结构。

第二个坑是视觉物件失真。  
地图、留声机、吧台、椅子这些元素，单独看似乎只是装饰，但它们其实决定了整个网页到底像不像一个咖啡馆。AI 很容易画出“差不多是那个意思”的东西，比如像讲台的吧台、像抽象拼图的留声机、像覆盖层一样的热点区域。这些问题如果不较真，页面仍然能用；但只要较真，就必须不断替换素材、调整比例、重设位置。这里我学到的是：视觉问题不能靠语言安慰自己，必须把“不像”明确说出来。

第三个坑是交互区域和实际视觉不一致。  
比如一开始咖啡杯的抽签交互用了较大的透明热点，看起来并不好，像网页上额外贴了一块东西；后来虽然加了提示，但又会影响画面完整性。最后我才逐渐意识到，交互区如果脱离视觉对象，就会破坏沉浸感。这也是我后来不断要求“交互点隐藏在图像位置”“不要虚空椭圆”“不要额外大框”的原因。

第四个坑是弹窗的内容组织。  
地图部分最开始虽然能弹出来，但地图和右边列表的关系不够清晰，滚动条虽然画出来了，却并没有真正形成独立滚动区域。这个问题很典型：从“代码上有滚动条”到“交互上真正可滚动”，中间其实差很多。后来通过限制高度、拆分滚动容器、把地图和列表分层，体验才变得顺畅。

第五个坑是部署与工程细节。  
在把项目部署到 Vercel 的过程中，我先遇到了 Git 用户信息未配置的问题，接着是远程仓库地址写错，又遇到了远程仓库已有内容导致 push 被拒绝。这些并不是页面本身的问题，但它们会影响作品能否真正上线。最终通过补全 Git 配置、重新设置 origin、再拉取合并历史，项目才顺利部署到公网。

## 六、我对 Vibe Coding 的理解

做完这次作业之后，我最大的感受是：AI 非常适合帮我快速搭出一个“可运行的第一版”，但真正决定作品质量的，仍然是我是否能持续提出清晰的判断。它不是“我想一个最终结果，然后 AI 一步到位做出来”，更像是“我先有一个方向，然后不断看结果、纠偏、重写要求、再推进下一轮”。在这个过程中，Prompt 不是咒语，而是设计判断的文字化。

我也更能理解老师为什么把“踩坑复盘”设成评分重点。因为真正有参与感的作业，不可能只有顺滑的一次生成；它一定会包括很多“不对”“不像”“太普通”“这个区域很丑”“这个交互破坏氛围”等等具体判断。也正是这些判断，构成了我对 AI 的主导。

如果让我重做一次，我会更早地把需求拆成三层：

1. 先确定页面结构，不急着堆内容。
2. 再确定视觉物件，不让抽象占位长期存在。
3. 最后再填真实数据和部署。

这样会比一开始边搭结构边塞内容更稳定。

总体来说，我认为这次作品最有价值的部分，不是“我做出了一个主页”，而是我学会了怎样和 AI 一起把“模糊的审美偏好”慢慢逼近成一个具体可交付的网页。这种能力比写出某一页代码本身更重要。
